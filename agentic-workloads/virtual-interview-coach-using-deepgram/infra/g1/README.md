# G1 infra

CloudFormation for the Real-Time Voice Interview Loop (Gate G1).

## `gate-enablement.yaml` — what the live gate run needs

Provisions **only** the cloud resources the local measurement harness needs to produce a
live SC-001 number — not the full production stack:

| Resource | Purpose | Feeds `.env` key |
|---|---|---|
| `AWS::Bedrock::Agent` + `AgentAlias` | AgentCore reply path | `AGENTCORE_AGENT_ID`, `AGENTCORE_AGENT_ALIAS_ID` |
| `AWS::IAM::Role` | Agent → InvokeModel permission | — |
| `AWS::RDS::DBInstance` (Postgres 16, **private, no public IP**) | 3-table latency slice | `DATABASE_URL` |
| `AWS::EC2::Subnet` x2 + route table | private subnets for the DB (no IGW/NAT route) | — |
| `AWS::EC2::SecurityGroup` | Postgres ingress (in-VPC sources only) | — |
| `AWS::Logs::LogGroup` | worker logs | — |
| `AWS::CloudWatch::Dashboard` | live `response_gap` p50/p95 vs gate lines | (`CW_METRIC_NAMESPACE`) |
| `AWS::SecretsManager::Secret` | Deepgram key for the *deployed* worker | — |

The full VPC / ECS Fargate / NLB / Cognito production stack is task **T048** and is
deliberately **not** here: the STT/TTS/transport seams still `raise NotImplementedError`
and no container images exist, so deploying it now would stand up infra that cannot serve
a session.

## Manual prerequisite (CloudFormation cannot do this)

Enable **Claude model access** in the Bedrock console → *Model access* in your target
region, **before** deploying. Without it the agent and the harness fail with `AccessDenied`.

## Post-create: disable RDS managed-secret rotation (do this once after the DB is created)

The RDS master password is RDS-managed and ships with auto-rotation **enabled (every 7 days)**.
For this demo we disable rotation entirely — the secret stays the durable home for the password
in Secrets Manager, it just doesn't rotate. The rotation schedule lives on the RDS-owned
`rds!db-*` secret (its ARN doesn't exist until RDS creates the DB), so this is an API step, not a
template property. Re-run it after any teardown + recreate:

```bash
SECRET=$(aws rds describe-db-instances --region us-west-2 \
  --db-instance-identifier interviewcoach-g1-latency \
  --query 'DBInstances[0].MasterUserSecret.SecretArn' --output text)
aws secretsmanager cancel-rotate-secret --region us-west-2 --secret-id "$SECRET"
# verify: RotationEnabled=false, NextRotationDate=null, the secret value unchanged
aws secretsmanager describe-secret --region us-west-2 --secret-id "$SECRET" \
  --query '{RotationEnabled:RotationEnabled,NextRotation:NextRotationDate}' --output json
```

Defense-in-depth (independent of the above): the services are also **rotation-proof in code** —
each task gets `DB_SECRET_ARN` + a `db-secret-read` grant, and `config._database_url` emits a
passwordless DSN so asyncpg fetches the live password from Secrets Manager **per connection** (via
`db_secret.make_password_provider`, re-evaluated on every new pooled connection). So even if
rotation is ever re-enabled, it stays transparent — no task restart, no sweep outage.

History: before the code fix (≤ tag `a33cc41`), tasks baked `DB_PASSWORD` in at container start
and never re-read it, so the 2026-06-09 rotation broke the report-worker's hourly retention sweep
for ~2 days (46 consecutive `InvalidAuthorizationSpecificationError`). Rotation is now off and the
code is rotation-proof, so neither recurs.

## Deploy

```bash
REGION=us-east-1
STACK=interviewcoach-g1

# Discover the VPC + its CIDR (default VPC is fine for a POC). The stack CREATES its own two
# private DB subnets inside this VPC — you do NOT pass subnet ids any more.
aws ec2 describe-vpcs --filters Name=isDefault,Values=true \
  --query 'Vpcs[0].{Id:VpcId,Cidr:CidrBlock}' --output table --region $REGION

aws cloudformation deploy \
  --stack-name $STACK \
  --template-file infra/g1/gate-enablement.yaml \
  --capabilities CAPABILITY_IAM \
  --region $REGION \
  --parameter-overrides \
    VpcId=<VPC_ID> \
    VpcCidr=<VPC_CIDR>            # e.g. 172.31.0.0/16 for the default VPC
    # PrivateSubnet1Cidr / PrivateSubnet2Cidr default to 172.31.240.0/24 + .241.0/24 (safe high
    # ranges in the default VPC). Override them if your VPC CIDR differs or those ranges are taken.
```

The RDS instance is **private** (`PubliclyAccessible: false`, in stack-created private subnets
with no IGW/NAT route) and its security group only accepts Postgres from **in-VPC** sources — it
is not reachable from the internet. The DB master password is generated by RDS and stored in
Secrets Manager (never in the template or in plaintext).

> **Applying onto an EXISTING (previously public) deployment:** `PubliclyAccessible` and the DB
> subnet group are **immutable**, so `cloudformation deploy`-ing this change forces a DB
> **replacement** — the data is reset. For this demo that is an accepted outcome (the DB is
> `DeletionPolicy: Delete` with no backups by design). Just deploy, then re-run the in-VPC schema
> migration below against the fresh instance. (If you ever need to preserve data, snapshot first
> and restore into the private subnet group instead.)

## Wire outputs back into `voice-worker/.env`

```bash
aws cloudformation describe-stacks --stack-name $STACK --region $REGION \
  --query 'Stacks[0].Outputs' --output table
```

Then set:

```
AGENTCORE_AGENT_ID=<AgentId output>
AGENTCORE_AGENT_ALIAS_ID=<AgentAliasId output>
```

The services build `DATABASE_URL` at runtime from the DB outputs + the RDS-managed secret
(`DB_SECRET_ARN` → passwordless DSN, fetched per connection). For ad-hoc inspection the secret
value is:

```bash
SECRET_ARN=<DBMasterSecretArn output>
aws secretsmanager get-secret-value --secret-id "$SECRET_ARN" --region $REGION \
  --query SecretString --output text
# -> {"username":"icadmin","password":"....", ...}
# DATABASE_URL form: postgres://icadmin:<password>@<DBEndpoint>:5432/interviewcoach
```

**The DB is private — your laptop cannot `psql` to it directly.** Apply the schema from
**inside the VPC**. The supported path is a one-off ECS task on the worker task def (in-VPC, no
SG/bastion dance — it honors `DB_SECRET_ARN`):

```bash
# Run the idempotent migration as an in-VPC ECS one-off task (see CLAUDE.md):
aws ecs run-task --cluster interviewcoach-g1-demo \
  --task-definition interviewcoach-g1-demo-voice-worker \
  --launch-type FARGATE \
  --network-configuration "awsvpcConfiguration={subnets=[<task-subnet>],securityGroups=[<task-sg>],assignPublicIp=ENABLED}" \
  --overrides '{"containerOverrides":[{"name":"voice-worker","command":["python","-m","src.db_migrate"]}]}' \
  --region $REGION
```

For G1 this creates `voice_session`, `conversation_turn`, `turn_latency`. For F002 (G2) the same
idempotent migration additively extends those tables and adds the product-model tables + pgvector
(the migration runs `CREATE EXTENSION IF NOT EXISTS vector;` itself). Alternatively, reach the DB
over an SSM port-forward / bastion in the same VPC if you need an interactive `psql`.

## F002 (G2) raw-resume storage — `ResumeBucket` + `ResumeKmsKey`

The stack now provisions a dedicated SSE-KMS bucket (`${STACK}-resumes-${ACCOUNT}`) and CMK
(`alias/${STACK}-resume`) for the raw resume file — the sole durable home for that PII
alongside RDS (Constitution III). The bucket has public access fully blocked and a 30-day
expiry lifecycle matching the default retention (FR-217); the `BackendTaskRole` is scoped to
PutObject/GetObject/DeleteObject on this bucket plus Encrypt/Decrypt on this key only. The
bucket name + key id reach the backend as `RESUME_BUCKET` / `RESUME_KMS_KEY_ID`. The Bedrock
invoke grant on the backend role is for OFF-clock work only (resume parsing + the one-time
JD embedding); nothing here is on the response_gap path.

## Cost & teardown

Smallest sensible footprint: `db.t4g.micro`, 20 GB gp3, no Multi-AZ, no backups
(`BackupRetentionPeriod: 0`). The Bedrock agent, log group, dashboard, and secret are
near-zero at rest; you pay per model invocation during a gate run. Tear everything down
with:

```bash
aws cloudformation delete-stack --stack-name $STACK --region $REGION
```

(`DeletionPolicy: Delete` on the DB, log group, resume bucket, and resume CMK means nothing
is retained on teardown. G1 stored no PII; F002 does store raw resume PII in `ResumeBucket`
under consent — teardown removes it with the stack, the bounded-blast-radius end state of
FR-219.)
