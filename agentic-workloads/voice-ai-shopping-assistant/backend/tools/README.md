# Tools & Gateway (`ToolsStack`)

**Deploys:** `ToolsStack` (tool Lambdas + AgentCore Gateway + MCP targets;
resources defined in `infra/lib/tools-stack.ts`).

## Build
One Python 3.12 arm64 Lambda per tool; each registered as a Gateway MCP target.
Lambdas reach Aurora via **Data API** (`rds-data:ExecuteStatement`) — no VPC, no pooling.
Read DB connection from SSM (`/aisle/db/*`).

### `search_products` — enhanced (lexical + semantic)
Free-text `query` is tokenized (multi-word, order-independent; per-word match on
name/brand) with `pg_trgm` relevance ranking, plus optional filters the LLM
fills from natural language: `dietary_tags` (all), `exclude_allergens` (none),
`min/max_price_cents` (effective price, special-aware), `quality_tier`,
`in_stock_only`, `on_special_only`, `sort`
(`relevance|price_asc|price_desc|savings_desc`), and `mode`
(`auto|lexical|semantic`). The Gateway target schema exposes the same params
(allowed values are in the descriptions — AgentCore Gateway schema does not
support `enum`).

**Semantic search:** products are embedded at seed time (Bedrock Cohere Embed
English v3, 1024-dim, stored in a pgvector `embedding` column with an HNSW cosine
index). `mode=auto` (default) runs lexical first and falls back to semantic when
keywords miss — so vague/conceptual queries ("something fizzy", "taco night",
"healthy breakfast") resolve **without** the prompt pre-translating them, while
precise lookups stay on the fast keyword path. The query is embedded at request
time (`input_type=search_query`); the Lambda needs `bedrock:InvokeModel` on the
Cohere model. The response carries `search_mode` (`lexical`|`semantic`). See
`agent/SEARCH_EXAMPLES.md` (§9) for utterance → call mappings.

Catalog: a synthetic Aisle catalogue of grocery products + specials, generated by
`db/seed/generate_catalogue.py` (fictional brands, deterministic uuid5 ids, no
real-retailer data or images).

### `create_order` + merchant API — payments & security
`create_order` runs a flag-gated x402 / AgentCore `ProcessPayment` leg against
`merchant_api` (a fake x402 merchant). The merchant is fronted by an
**IAM-authorized API Gateway REST API** (`aisle-merchant-api`, `POST /`,
exported as SSM `/aisle/merchant/url`) — **not** a public Lambda Function URL.
Only `create_order`'s role is granted `execute-api:Invoke`, and `create_order`
**SigV4-signs** its POST (service `execute-api`); unsigned requests get 403.
This replaced an earlier public Function URL (`AuthType: NONE`) flagged as an
open-Lambda-policy finding. `create_order` vendors a newer `boto3`
(`requirements.txt`) for `process_payment` + signing.
