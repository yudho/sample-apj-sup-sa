# batch — vLLM on AWS Batch

Deploy any open-source LLM onto **AWS Batch** for high-throughput batch
inference. Each Batch job runs a container that starts **vLLM locally** and
drives it with **asyncio concurrency**, so continuous batching saturates the
GPU — without the operational cost of a long-running endpoint.

Lineup currently shipped (all under `models/`):

* `qwen3_8b` — `Qwen/Qwen3-8B` (Apache-2.0)
* `mistral_small_3_2_24b` — `mistralai/Mistral-Small-3.2-24B-Instruct-2506` (Apache-2.0)
* `qwen3_30b_a3b` — `Qwen/Qwen3-30B-A3B-Instruct-2507` (Apache-2.0, MoE)
* `gemma_4_31b` — `google/gemma-4-31B-it` (Apache-2.0)
* `medgemma_27b` — `google/medgemma-27b-text-it` (HAI-DEF, gated)
* `llama_4_scout_17b` — `meta-llama/Llama-4-Scout-17B-16E-Instruct` (Llama-4, gated)

Extending to a new model is a per-folder operation under `models/`.

## What this repo gives you

1. **Declarative plans.** `BatchDeploymentPlan` captures the instance type,
   TP/DP/PP parallelism, capacity mode (spot / on-demand / ODCR / ML
   Capacity Block), and queue topology.
2. **CloudFormation deployment.** `deploy(plan)` translates the plan into
   a complete CFN stack — IAM, staging S3 bucket, ECR repo, Batch compute
   environments + queues + job definition — and applies it via boto3.
3. **Container runtime.** A Dockerfile (`src/llm_batch_deploy/runtime/`)
   starts vLLM as a subprocess, then runs an asyncio driver that
   semaphore-bounds concurrent HTTP requests.
4. **Invoker.** `submit_batch(input_sources=[...], ...)` uploads local
   inputs to S3, writes JSONL manifests, and SubmitJobs one Batch job
   per shard. Returns a DataFrame report.
5. **Waiter + collector.** `wait_for_completion(report)` polls
   DescribeJobs in 100-job chunks; `download_outputs(report, dir)` pulls
   results locally; `sample_outputs(report, n=3)` peeks at records.
6. **Notebooks.** `notebooks/<model>_batch.ipynb` wires it all together
   end-to-end. One notebook per model in the lineup, all generated by
   `scripts/build_notebook.py --all`.

## Quick start

```bash
# 1. Create venv + install
./scripts/setup_env.sh
source .venv/bin/activate

# 2. Run tests
pytest tests/

# 3. Launch JupyterLab
./scripts/start_jupyter.sh
# Open notebooks/<model>_batch.ipynb (e.g. qwen3_8b_batch.ipynb)
# kernel = Python 3 (ipykernel).
```

The notebook covers: deploy → (one-time) image build → upload inputs →
submit → wait → sample → download → teardown.

## Concurrency model (the important bit)

An AWS Batch "job" is **not** an inference call. One Batch job =
**one shard of many inference calls**:

```
┌──────────────────────────────────────────────────────────────┐
│ One container (one Batch job)                                │
│                                                              │
│   vLLM server (port 8000, continuous batching) ◄────┐        │
│         ▲                                           │        │
│         │ HTTP POST /v1/chat/completions            │        │
│         │ (up to `in_flight_per_job` in flight)     │        │
│                                                              │
│   asyncio driver (semaphore-bounded) ───────────────┘        │
│         ▲                                                    │
│         │ yields records from:                               │
│                                                              │
│   S3 input files listed in manifest                          │
└──────────────────────────────────────────────────────────────┘
```

Batch scales by running **many containers in parallel** (one per queued
job). vLLM's continuous batching fuses concurrent requests into GPU
passes — the driver's only job is to keep the queue non-empty.

Default `in_flight_per_job = 32` is safe for 27B-class models. Tune this
per-model against your workload — a single-container endpoint benchmark
(e.g. [LLMeter](https://github.com/awslabs/llmeter)) is the usual way to
find the sweet spot for latency vs throughput.

## Adding a new model

Mirror any of the existing per-model folders under `models/` (e.g.
`models/qwen3_8b/`):

1. `model_spec.py` — one `ModelSpec(resource_prefix="my-model", ...)`.
2. `batch_plans.py` — one or more factory functions returning
   `BatchDeploymentPlan` with your chosen instance type + parallelism.
3. Add an entry to `scripts/build_notebook.py`'s `MODEL_CONFIGS`
   registry, then `build_notebook.py --model my_model` to emit the
   notebook.
4. `from models.my_model import my_plan; stack = deploy(my_plan())`.

The framework code doesn't need to change.

## Requirements

* Python 3.11
* AWS account with Batch + EC2 + ECR + S3 + IAM permissions
* Hugging Face token (for gated models — MedGemma-27B, Llama-4-Scout)
* Docker (for building the container image)
